Surface reasoning/thinking on the foreground spinner, extract a shared spinner primitive, and render the tool-pending `Executing...` indicator by default so consumers without custom tool renderers (e.g. `minimal-agent`) see it automatically.

- TUI: while the model streams reasoning parts, the foreground spinner now stays visible with the label swapped to `Thinking...`. Reasoning parts are treated as non-visible for the first-visible-part trigger, so the spinner is only cleared when real text/tool output arrives. `PiTuiStreamState` gains optional `onReasoningStart` / `onReasoningEnd` callbacks; a new `handleReasoningEnd` handler fires `onReasoningEnd` (previously `reasoning-end` was silently dropped via `IGNORE_PART_TYPES`). `renderAgentStream` wires these into the spinner so the label temporarily becomes `Thinking...` during reasoning spans and restores the caller-provided base label (`Working...`) when reasoning ends.
- TUI: fix `Thinking...` label missing on the second reasoning span of a turn (post tool-call). When the continuation stream emits a visible part (`text-start`, `tool-result`, etc.) before the `reasoning-start`, `clearStreamingLoader` nulls the foreground spinner first, so the subsequent `foregroundStatus?.setMessage("Thinking...")` silently no-ops. `onReasoningStart` now revives the spinner via `showLoader("Thinking...")` if it was cleared, and `onReasoningEnd` tears down only the spinner it revived (so ordinary flows that kept the base `Working...` loader alive still restore that label unchanged).
- TUI: new shared primitive `@ai-sdk-tool/tui` → `pending-spinner.ts` exposing `PENDING_SPINNER_FRAMES`, `PENDING_SPINNER_INTERVAL_MS`, `stylePendingIndicator(frame, message)`, and `createSpinnerTicker(onFrame, options?)`. `StatusSpinner`, `FooterStatusBar`, `BaseToolCallView`'s tool-pending block, and the internal `pi-tui-stream-renderer.ts` tool view in CEA all route through this primitive, so any future palette/cadence/glyph change lands in one place.
- TUI: `BaseToolCallView` now renders the `Executing...` pending indicator automatically during the raw-fallback path (when no custom `ToolRendererMap` handles the tool). The indicator is appended below the tool block while the call is in flight — i.e. input has streamed but no output/error/denial has arrived — and is removed as soon as any terminal state lands. Consumers that already provide custom pretty renderers are unaffected; consumers like `minimal-agent` get the spinner with no code changes.
- TUI: fix the inline `Executing...` indicator rendering one blank line below the tool block instead of immediately under it. `BaseToolCallView.content` is switched from a plain `Markdown` to the existing `TrimmedMarkdown` (which strips trailing blank lines from markdown output). The trailing `""` came from the `space` token that pi-tui's `Markdown.render()` emits between blocks when `blocks.join("\n\n")` leaves a terminal `\n\n` sequence; since `TrimmedMarkdown` was already used for `readHeader` in pretty blocks, this just reuses the same primitive. New regression test locks in that the spinner sits on the line immediately below the tool block with no extra blank gap.
- TUI: `BaseToolCallView`'s existing pretty-block `Executing..` indicator adopts the canonical braille frames (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`) with cyan-frame / dim-message styling, and the label is normalized to `Executing...` (three dots, matching `Processing...` / `Working...` / `Thinking...`). Previously tool-pending blocks used plain ASCII `- \ | /` frames with no color.
- CEA: the internal `pi-tui-stream-renderer.ts` pending spinner is kept in sync via the shared primitive. Tests updated accordingly (`EXECUTING_SPINNER_TEXT_REGEX` matches braille frames; string assertions updated to `Executing...`). New `tool-call-view.test.ts` covers the automatic fallback indicator (appears while pending, cleared on output/error, never rendered before any input arrives).
